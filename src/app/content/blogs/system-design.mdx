## 1. What is System Design? {#what-is-system-design}

System design is the process of defining the **architecture, components, modules, interfaces, and data flows** of a system to satisfy a set of specified requirements.

As a full-stack developer, you already build systems — but system design takes it further: you're no longer asking *"how do I build this feature?"* but *"how do I build this so it works for 10 million users, never goes down, and costs a reasonable amount to run?"*

### Why It Matters

| Scale | Problem You're Solving |
|---|---|
| 1–1,000 users | Correctness and features |
| 1,000–100,000 users | Performance bottlenecks |
| 100,000–10M users | Scalability and reliability |
| 10M+ users | Distributed systems, fault tolerance, cost |

### The Two Dimensions of System Design

**Functional Requirements** — What the system *does*:
- Users can upload photos
- Users can send messages in real-time
- Search results return within 200ms

**Non-Functional Requirements** — How well the system *performs*:
- **Availability:** 99.99% uptime (52 minutes of downtime per year)
- **Scalability:** Handle 10x traffic spikes
- **Consistency:** Data is never lost or corrupted
- **Latency:** P99 response time under 500ms

## 2. The Building Blocks {#the-building-blocks}

Before designing any system, you must be fluent in the fundamental components that compose every large-scale architecture.

### DNS (Domain Name System)

DNS translates human-readable domain names (`www.yourportfolio.dev`) into machine-routable IP addresses. It is the phone book of the internet.

**Flow:**
```
User types URL → Browser checks local cache → ISP DNS resolver →
Root nameserver → TLD nameserver → Authoritative nameserver → IP returned
```

**Example:** When you deploy your Next.js portfolio on Vercel and configure a custom domain, you're updating DNS A-records to point your domain at Vercel's load balancer IP.

---

### Servers and Clients

```
Client (Browser / Mobile App)
        │
        │  HTTP/HTTPS Request
        ▼
    Web Server (Express, Nginx, Next.js)
        │
        │  Internal RPC / API Call
        ▼
  Application Server (Business Logic)
        │
        │  Query
        ▼
    Database / Cache
```

---

### IP Addresses and Ports

Every server has an IP address. Services listen on ports:
- `80` — HTTP
- `443` — HTTPS
- `5432` — PostgreSQL
- `6379` — Redis
- `27017` — MongoDB

When you run `docker run -p 8080:3000 myapp`, you're binding host port `8080` to container port `3000` — the same port binding concept applies at cloud infrastructure scale.

---

### Latency Numbers Every Engineer Must Know

| Operation | Latency |
|---|---|
| L1 cache reference | 0.5 ns |
| RAM access | 100 ns |
| SSD random read | 150 µs |
| Network round trip (same datacenter) | 500 µs |
| Network round trip (cross-continent) | 150 ms |
| HDD seek | 10 ms |
| Reading 1 MB from SSD | 1 ms |

**Takeaway:** A database call is ~1000× slower than a memory access. This is why **caching** is one of the highest-leverage optimizations in system design.

## 3. Scalability {#scalability}

Scalability is the system's ability to handle increasing load by adding resources.

### Vertical Scaling (Scale Up)

Add more power to a single machine: more CPU cores, more RAM, faster SSD.

```
Before:  [Server: 4 CPU, 16GB RAM]
After:   [Server: 32 CPU, 128GB RAM]
```

**Pros:** Simple, no code changes, no distributed system complexity.

**Cons:** Has a hard ceiling (you can't infinitely upgrade one machine). Single point of failure. Expensive.

**Example:** A PostgreSQL database handling 500 concurrent connections gets upgraded from `db.t3.medium` to `db.r6g.4xlarge` on AWS RDS.

---

### Horizontal Scaling (Scale Out)

Add more machines. Distribute the load across multiple servers.

```
Before:  [Server 1]

After:   [Server 1]
         [Server 2]
         [Server 3]
              ▲
         [Load Balancer] ← All traffic enters here
```

**Pros:** Theoretically unlimited scale. No single point of failure. Cost-effective with commodity hardware.

**Cons:** Requires stateless application design. Distributed system complexity. Session and cache management becomes non-trivial.

**Example:** Your Node.js API is deployed as 3 replicas in a Kubernetes cluster. The load balancer distributes incoming requests across all three.

---

### Stateless vs Stateful Architecture

For horizontal scaling to work, your application servers must be **stateless** — they cannot store user session data in local memory.

**❌ Stateful (breaks horizontal scaling):**
```javascript
// Session stored in-memory on Server 1
app.post('/login', (req, res) => {
  sessions[req.body.userId] = { loggedIn: true };
  // If next request goes to Server 2, session is gone!
});
```

**✅ Stateless (scales horizontally):**
```javascript
// Session stored in Redis — accessible by all servers
app.post('/login', async (req, res) => {
  await redis.set(`session:${req.body.userId}`, JSON.stringify({ loggedIn: true }));
  res.json({ token: generateJWT(req.body.userId) });
});
```

## 4. Load Balancing {#load-balancing}

A Load Balancer is a server that sits in front of your application servers and distributes incoming traffic across them.

```
                    ┌─────────────┐
Incoming Traffic ──▶│ Load Balancer│
                    └──────┬──────┘
              ┌────────────┼────────────┐
              ▼            ▼            ▼
         [Server 1]   [Server 2]   [Server 3]
```

### Load Balancing Algorithms

**Round Robin** — Requests are distributed sequentially across servers.
```
Request 1 → Server 1
Request 2 → Server 2
Request 3 → Server 3
Request 4 → Server 1 (cycle repeats)
```
Best for: Servers with uniform capacity and request cost.

---

**Weighted Round Robin** — Servers with more capacity receive proportionally more traffic.
```
Server 1 (weight: 3) → receives 3 of every 5 requests
Server 2 (weight: 2) → receives 2 of every 5 requests
```

---

**Least Connections** — New requests go to the server with the fewest active connections.

Best for: Long-lived connections (WebSockets, file uploads) where servers may become unevenly loaded.

---

**IP Hash** — The client's IP address determines which server handles the request (consistent routing).

Best for: Stateful applications that cannot yet be fully migrated to external session stores.

---

### Layer 4 vs Layer 7 Load Balancing

| Type | Operates At | Routing Decision Based On | Example |
|---|---|---|---|
| L4 (Transport) | TCP/UDP | IP address and port | AWS NLB |
| L7 (Application) | HTTP/HTTPS | URL path, headers, cookies | AWS ALB, Nginx |

**Example — L7 Load Balancing with Nginx:**
```nginx
upstream api_servers {
    server api-1.internal:3000;
    server api-2.internal:3000;
    server api-3.internal:3000;
}

server {
    listen 80;

    location /api/ {
        proxy_pass http://api_servers;
    }

    location /static/ {
        root /var/www/static;
    }
}
```

---

### Health Checks

Load balancers continuously probe backend servers. If a server fails a health check, traffic is automatically rerouted to healthy instances.

```
Load Balancer → GET /health → Server 1 → 200 OK ✅ (in rotation)
Load Balancer → GET /health → Server 2 → 503     ❌ (removed from rotation)
```

## 5. Caching {#caching}

Caching stores frequently accessed data in fast-access storage to reduce latency and database load.

**The Fundamental Principle:** Compute once, serve many times.

```
Without Cache:  Request → DB Query (50ms) → Response
With Cache:     Request → Cache Hit (1ms)  → Response
```

### Cache Hit vs Cache Miss

- **Cache Hit:** The requested data exists in the cache. Served immediately.
- **Cache Miss:** The data is not in the cache. Fetch from the source, store in cache, return to client.

### Caching Strategies

**Cache-Aside (Lazy Loading)** — The application manages the cache explicitly.

```javascript
async function getUserProfile(userId) {
  // 1. Check cache first
  const cached = await redis.get(`user:${userId}`);
  if (cached) return JSON.parse(cached); // Cache HIT

  // 2. Cache MISS — fetch from database
  const user = await db.query('SELECT * FROM users WHERE id = $1', [userId]);

  // 3. Populate cache for future requests (TTL: 1 hour)
  await redis.setex(`user:${userId}`, 3600, JSON.stringify(user));

  return user;
}
```

**Best for:** Read-heavy workloads. Only requested data is cached (no wasted memory).

---

**Write-Through** — Every write to the database also synchronously writes to the cache.

```javascript
async function updateUserProfile(userId, data) {
  // Write to DB and cache simultaneously
  await db.query('UPDATE users SET name = $1 WHERE id = $2', [data.name, userId]);
  await redis.setex(`user:${userId}`, 3600, JSON.stringify(data));
}
```

**Best for:** Read-heavy workloads where cache consistency is critical.

---

**Write-Behind (Write-Back)** — Writes go to cache first, asynchronously flushed to the database later.

**Best for:** Write-heavy workloads where brief inconsistency is acceptable (analytics counters, view counts).

---

### Cache Eviction Policies

When the cache reaches capacity, it must decide what to remove.

| Policy | Strategy | Best For |
|---|---|---|
| **LRU** (Least Recently Used) | Evict the item that was accessed longest ago | General purpose |
| **LFU** (Least Frequently Used) | Evict the item accessed the fewest times | Long-lived access pattern data |
| **TTL** (Time-To-Live) | Evict items after a fixed expiration time | Session data, short-lived tokens |
| **FIFO** | Evict the oldest inserted item | Simple queues |

---

### Where to Cache

**Client-Side:** Browser cache, Service Workers (great for your Next.js portfolio's static assets).

**CDN Cache:** Geographically distributed edge caches (Cloudflare, Vercel Edge Network).

**Server-Side (Application Cache):** In-process memory (node-cache), or external (Redis, Memcached).

**Database Cache:** Query result cache (MySQL Query Cache, PostgreSQL `pg_prewarm`).

---

### Cache Stampede (Thundering Herd)

A critical failure mode: a popular cache key expires simultaneously for thousands of concurrent requests. All requests miss the cache and hammer the database at once.

**Solution — Probabilistic Early Expiration:**
```javascript
async function getWithJitter(key, ttl, fetchFn) {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  // Add random jitter to TTL to desynchronize expiration
  const jitter = Math.random() * 300; // 0–5 minutes
  const data = await fetchFn();
  await redis.setex(key, ttl + jitter, JSON.stringify(data));
  return data;
}
```

## 6. Databases — SQL vs NoSQL {#databases}

### SQL (Relational Databases)

Stores data in structured tables with predefined schemas. Relationships between entities are modeled via foreign keys.

**Examples:** PostgreSQL, MySQL, SQLite, CockroachDB

```sql
-- Relational schema: Users and their Posts
CREATE TABLE users (
  id         SERIAL PRIMARY KEY,
  email      VARCHAR(255) UNIQUE NOT NULL,
  name       VARCHAR(100),
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE posts (
  id         SERIAL PRIMARY KEY,
  user_id    INTEGER REFERENCES users(id) ON DELETE CASCADE,
  title      VARCHAR(500),
  content    TEXT,
  created_at TIMESTAMP DEFAULT NOW()
);

-- JOIN query: fetch all posts by a user
SELECT u.name, p.title, p.created_at
FROM users u
JOIN posts p ON p.user_id = u.id
WHERE u.email = 'dev@example.com'
ORDER BY p.created_at DESC;
```

**ACID Properties:**
- **Atomicity:** A transaction either fully completes or fully rolls back. No partial writes.
- **Consistency:** The database transitions from one valid state to another.
- **Isolation:** Concurrent transactions don't interfere with each other.
- **Durability:** Committed transactions are persisted even after a crash.

---

### NoSQL Databases

Schema-flexible, horizontally scalable databases optimized for specific data access patterns.

**Document Stores (MongoDB, Firestore)**
```javascript
// Flexible schema — no migrations needed for new fields
db.users.insertOne({
  _id: ObjectId(),
  email: "dev@example.com",
  name: "Rudra",
  skills: ["React", "Node.js", "Docker"],
  address: {
    city: "Bhubaneswar",
    state: "Odisha"
  }
});
```
Best for: Content management, user profiles, catalogs, JSON-heavy APIs.

---

**Key-Value Stores (Redis, DynamoDB)**
```bash
SET session:user123 '{"userId":"123","role":"admin"}' EX 3600
GET session:user123
```
Best for: Sessions, caching, leaderboards, pub/sub.

---

**Wide-Column Stores (Apache Cassandra, HBase)**

Optimized for massive write throughput and time-series data. Used by Instagram, Netflix, and Discord for storing billions of events.

Best for: Analytics, activity feeds, IoT data streams, time-series.

---

**Graph Databases (Neo4j, Amazon Neptune)**

Optimized for traversing complex relationships between entities.

Best for: Social networks (friend-of-a-friend), recommendation engines, fraud detection.

---

### SQL vs NoSQL — Decision Framework

| Factor | Choose SQL | Choose NoSQL |
|---|---|---|
| Data structure | Structured, relational | Flexible, hierarchical, or unstructured |
| Query complexity | Complex joins and aggregations | Simple lookups by key |
| Consistency requirement | Strong ACID | Eventual consistency acceptable |
| Scale pattern | Vertical | Horizontal |
| Schema | Stable, well-defined | Evolving, uncertain |
| Examples | Banking, e-commerce orders | Social feeds, analytics, caching |

## 7. Database Sharding and Replication {#database-sharding-and-replication}

### Replication

Replication maintains synchronized copies of your database across multiple servers.

**Primary-Replica (Master-Slave) Replication:**
```
All WRITES  ──▶  [Primary DB]
                      │
              (async replication)
                 ┌────┴────┐
                 ▼         ▼
          [Replica 1] [Replica 2]
               ▲         ▲
All READS ─────┴─────────┘
```

**Benefits:**
- **Read scalability:** Distribute read queries across replicas.
- **High availability:** If the primary fails, promote a replica.
- **Disaster recovery:** Replicas in separate geographic regions.

**Replication Modes:**
- **Synchronous:** Primary waits for replica acknowledgment before confirming the write. Strong consistency, higher latency.
- **Asynchronous:** Primary confirms write immediately; replicas catch up shortly after. Low latency, risk of replica lag.

---

### Database Sharding (Horizontal Partitioning)

Sharding splits a large dataset across multiple independent database instances, each holding a subset of the data (a **shard**).

```
Total Users: 100 million

Shard 1: Users A–M  (users where name starts A–M)
Shard 2: Users N–Z  (users where name starts N–Z)
```

**Shard Key Selection** is the most critical decision in sharding. A poor shard key creates **hot spots** (one shard receiving disproportionate traffic).

**Common Sharding Strategies:**

*Hash-Based Sharding:*
```javascript
function getShard(userId, totalShards) {
  return hash(userId) % totalShards;
}
// User ID 12345 → hash(12345) % 4 → Shard 2
```
Pros: Even distribution. Cons: Hard to add shards (requires rehashing).

*Range-Based Sharding:*
```
Shard 1: user_id 1        – 10,000,000
Shard 2: user_id 10000001 – 20,000,000
Shard 3: user_id 20000001 – 30,000,000
```
Pros: Simple range queries. Cons: Hot spots if IDs are sequential (new users always hit the last shard).

*Geographic Sharding:*
```
Shard US-EAST:  Users from North America
Shard EU-WEST:  Users from Europe
Shard AP-SOUTH: Users from Asia-Pacific
```
Pros: Low latency for regional users. Cons: Uneven growth if regions scale at different rates.

## 8. CAP Theorem {#cap-theorem}

The CAP Theorem states that a distributed data store can only **simultaneously guarantee two of three properties**:

```
           Consistency (C)
               /\
              /  \
             /    \
            /  CP  \
           /        \
          /    CA    \
         ──────────────
Availability (A) ──── Partition Tolerance (P)
                  AP
```

- **Consistency (C):** Every read receives the most recent write or an error.
- **Availability (A):** Every request receives a response (not necessarily the most recent data).
- **Partition Tolerance (P):** The system continues operating even if network partitions cause some nodes to be unreachable.

**In practice: Partition Tolerance is non-negotiable** in any real distributed system (networks fail). So the real choice is **CP vs AP**.

| System Type | Choice | Examples | Trade-off |
|---|---|---|---|
| CP | Consistency + Partition Tolerance | HBase, Zookeeper, etcd | May reject reads/writes during a partition |
| AP | Availability + Partition Tolerance | Cassandra, DynamoDB, CouchDB | May return stale data during a partition |

**Real-World Example:**

A banking transaction system must be **CP** — it's unacceptable for your account balance to show different values on two different ATMs simultaneously.

A social media "likes" counter can be **AP** — it's acceptable if the like count shows 10,249 on one server and 10,251 on another for a few seconds before converging.

## 9. Message Queues and Async Communication {#message-queues}

Message queues decouple producers (services that generate work) from consumers (services that process work), enabling **asynchronous, fault-tolerant communication**.

```
Without Queue:
[Service A] ──── synchronous HTTP call ────▶ [Service B]
             ← Service A blocked waiting ─

With Queue:
[Service A] ──▶ [Message Queue] ──▶ [Service B]
             returns immediately     processes at its own pace
```

### Core Concepts

- **Producer:** Publishes messages to the queue.
- **Consumer:** Reads and processes messages from the queue.
- **Queue:** The buffer that holds messages until consumed.
- **Dead Letter Queue (DLQ):** Holds messages that repeatedly fail processing for manual inspection.

### Real-World Example — Image Upload Pipeline

```javascript
// Producer: API Server receives upload request
app.post('/upload', async (req, res) => {
  const fileKey = await s3.upload(req.file);

  // Publish async job — API returns immediately
  await queue.publish('image-processing', {
    fileKey,
    userId: req.user.id,
    operations: ['resize', 'thumbnail', 'watermark']
  });

  res.json({ status: 'processing', fileKey });
});

// Consumer: Image Processing Worker (separate service)
queue.subscribe('image-processing', async (message) => {
  const { fileKey, userId, operations } = message;
  for (const op of operations) {
    await imageProcessor.run(op, fileKey);
  }
  await db.updateImageStatus(fileKey, 'complete');
  await notificationService.send(userId, 'Your image is ready!');
});
```

---

### Message Queue Patterns

**Point-to-Point (Queue):** One producer, one consumer. Each message processed exactly once.
```
Producer → [Queue] → Consumer A
```

**Publish-Subscribe (Topic):** One producer, multiple consumers. Each subscriber receives every message.
```
Producer → [Topic] → Consumer A (Email Service)
                  → Consumer B (Analytics Service)
                  → Consumer C (Push Notification Service)
```

---

### Popular Message Brokers

| Tool | Best For | Key Feature |
|---|---|---|
| **Redis Pub/Sub** | Simple, low-latency events | Already in your stack |
| **RabbitMQ** | Complex routing, enterprise patterns | Flexible exchange types |
| **Apache Kafka** | High-throughput event streaming | Log-based, message replay |
| **AWS SQS** | Managed, serverless queues | Zero ops overhead |
| **BullMQ** (Redis-backed) | Node.js job queues | Native JS, great DX |

## 10. API Design — REST, GraphQL, gRPC {#api-design}

### REST (Representational State Transfer)

The dominant API paradigm for web services. Resources are exposed at URLs; HTTP verbs define operations.

```
GET    /api/users              → List all users
GET    /api/users/:id          → Get a specific user
POST   /api/users              → Create a new user
PUT    /api/users/:id          → Replace a user
PATCH  /api/users/:id          → Partially update a user
DELETE /api/users/:id          → Delete a user
```

**REST API Design Best Practices:**
```javascript
// ✅ Good: Noun-based, versioned, consistent HTTP status codes
router.get('/v1/posts/:id', async (req, res) => {
  try {
    const post = await Post.findById(req.params.id);
    if (!post) return res.status(404).json({ error: 'Post not found' });
    res.status(200).json({ data: post });
  } catch (err) {
    res.status(500).json({ error: 'Internal server error' });
  }
});

// ❌ Bad: Verb-based, unversioned, inconsistent responses
router.get('/getPostById', (req, res) => {
  const post = Post.findById(req.query.id);
  res.send(post); // No error handling, no status code discipline
});
```

---

### GraphQL

A query language and runtime where the client specifies exactly what data it needs.

**The Over-fetching Problem GraphQL Solves:**
```graphql
# REST: GET /api/users/123 returns ALL user fields
# (name, email, phone, address, createdAt, billingInfo, preferences...)

# GraphQL: Client asks for ONLY what it needs
query {
  user(id: "123") {
    name
    email
    posts(last: 3) {
      title
      publishedAt
    }
  }
}
```

**GraphQL Schema Definition:**
```graphql
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  content: String!
  author: User!
  publishedAt: String!
}

type Query {
  user(id: ID!): User
  posts(limit: Int, offset: Int): [Post!]!
}

type Mutation {
  createPost(title: String!, content: String!): Post!
}
```

**Best for:** Mobile applications with bandwidth constraints, complex object graphs, rapid frontend iteration.

---

### gRPC

A high-performance RPC framework using Protocol Buffers (protobuf) for binary serialization. Primarily used for internal service-to-service communication.

```protobuf
// service.proto
syntax = "proto3";

service UserService {
  rpc GetUser (GetUserRequest) returns (UserResponse);
  rpc ListUsers (ListUsersRequest) returns (stream UserResponse);
}

message GetUserRequest {
  string user_id = 1;
}

message UserResponse {
  string id = 1;
  string name = 2;
  string email = 3;
}
```

| Dimension | REST | GraphQL | gRPC |
|---|---|---|---|
| Protocol | HTTP/1.1 | HTTP/1.1 | HTTP/2 |
| Payload | JSON (text) | JSON (text) | Protobuf (binary) |
| Performance | Moderate | Moderate | High |
| Browser support | Native | Native | Requires proxy |
| Best for | Public APIs | Complex queries | Internal microservices |

## 11. Microservices vs Monolith {#microservices-vs-monolith}

### Monolithic Architecture

The entire application — UI, business logic, data access — is deployed as a single, unified codebase.

```
┌────────────────────────────────────┐
│            Monolith                │
│  ┌──────────┐  ┌───────────────┐   │
│  │   Auth   │  │  User Profile │   │
│  └──────────┘  └───────────────┘   │
│  ┌──────────┐  ┌───────────────┐   │
│  │  Posts   │  │  Notifications│   │
│  └──────────┘  └───────────────┘   │
└──────────────────┬─────────────────┘
                   │
              [Single DB]
```

**Pros:** Simple to develop, test, deploy, and debug in early stages. No distributed system overhead.

**Cons:** As the codebase grows, deployments become risky, teams step on each other, and scaling one component requires scaling everything.

---

### Microservices Architecture

The application is decomposed into small, independently deployable services, each owning its domain and its data.

```
[API Gateway]
      │
      ├──▶ [Auth Service]         → [Auth DB]
      ├──▶ [User Service]         → [User DB]
      ├──▶ [Post Service]         → [Post DB]
      ├──▶ [Notification Service] → [Message Queue]
      └──▶ [Search Service]       → [Elasticsearch]
```

**Real Example — Twitter's Architecture Evolution:**

Twitter began as a Rails monolith. As it grew to hundreds of millions of users, it decomposed into services: Tweet Service, Timeline Service, Social Graph Service, Notification Service — each independently scalable and deployable.

---

### When to Choose Which

| Factor | Monolith | Microservices |
|---|---|---|
| Team size | Small (1–10 engineers) | Large (10+ engineers, multiple teams) |
| Product stage | Early-stage, MVP | Established, growing |
| Deployment frequency | Occasional | Continuous deployment |
| Domain complexity | Simple | Complex, multiple bounded contexts |
| Ops maturity | Low | High (Kubernetes, service mesh, tracing) |

> **Pragmatic Advice for Full-Stack Developers:** Start with a well-structured monolith with clear module boundaries. Extract services only when you feel a specific module's scaling or deployment requirements diverge from the rest. Premature microservices are a leading cause of unnecessary engineering complexity.

## 12. Content Delivery Networks (CDN) {#cdn}

A CDN is a geographically distributed network of proxy servers that caches content close to end users, dramatically reducing latency for static assets.

```
Without CDN:
User (India) ──── 200ms ────▶ Origin Server (US-East)

With CDN:
User (India) ──── 10ms ────▶ CDN Edge Node (Mumbai) ──── (cached)
```

### How CDN Caching Works

1. User in Mumbai requests `https://yoursite.com/hero-image.png`
2. CDN edge node in Mumbai checks its cache → **MISS**
3. Edge node fetches the asset from your origin server (US-East) → **200ms round trip**
4. Edge node caches the asset and serves it to the user
5. Next user in Mumbai requests the same asset → **HIT** → served in ~10ms

### Cache-Control Headers

You control CDN caching behavior via HTTP headers:

```javascript
// Express.js — Cache static assets for 1 year (fingerprinted files)
app.use('/static', express.static('public', {
  maxAge: '1y',
  immutable: true,
  setHeaders: (res, path) => {
    res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
  }
}));

// Cache API responses for 60 seconds
app.get('/api/trending-posts', (req, res) => {
  res.setHeader('Cache-Control', 'public, max-age=60, stale-while-revalidate=300');
  res.json(trendingPosts);
});
```

### What to Cache on a CDN

| Asset Type | Cache Duration | Strategy |
|---|---|---|
| Static JS/CSS bundles (hashed filenames) | 1 year | Immutable |
| Images (with content hash) | 1 year | Immutable |
| HTML pages | 60–300 seconds | Stale-while-revalidate |
| API responses (public, non-personalized) | 30–60 seconds | Short TTL |
| Personalized data | Never | No-store |

## 13. Rate Limiting {#rate-limiting}

Rate limiting controls the number of requests a client can make to your API within a time window, protecting against abuse, DDoS attacks, and resource exhaustion.

### Rate Limiting Algorithms

**Fixed Window Counter:**
```
Window: 60 seconds
Limit: 100 requests per window

User A: Made 100 requests at 00:59 ✅
User A: Makes request at 01:00 — new window resets ✅ (potential burst at boundary)
```

**Sliding Window Log:**
Tracks exact timestamps of every request. More accurate but memory-intensive.

**Token Bucket (Most Common):**
```
Bucket capacity: 100 tokens
Refill rate: 10 tokens/second

User makes request → consumes 1 token
Token count: 100 → 99 → 98...
If token count = 0 → reject with 429 Too Many Requests
```

**Implementation with Redis:**
```javascript
const rateLimit = async (userId, limit, windowSeconds) => {
  const key = `rate:${userId}`;
  const current = await redis.incr(key);

  if (current === 1) {
    // First request in window — set expiry
    await redis.expire(key, windowSeconds);
  }

  if (current > limit) {
    throw new Error('Rate limit exceeded'); // HTTP 429
  }

  return { remaining: limit - current };
};

// Express middleware
app.use('/api', async (req, res, next) => {
  try {
    const { remaining } = await rateLimit(req.ip, 100, 60);
    res.setHeader('X-RateLimit-Remaining', remaining);
    next();
  } catch (e) {
    res.status(429).json({
      error: 'Too Many Requests',
      retryAfter: 60
    });
  }
});
```

## 14. Consistent Hashing {#consistent-hashing}

Consistent hashing solves a fundamental distributed systems problem: **how do you add or remove servers in a cluster without remapping all data?**

### The Problem with Naive Hashing

With 3 servers and hash-based routing:
```
server = hash(request_key) % 3
```

If you add a 4th server:
```
server = hash(request_key) % 4
```

Nearly every request maps to a different server. All cached data is invalidated simultaneously. All database connections are rerouted. **Catastrophic.**

---

### The Consistent Hashing Solution

Imagine a ring of hash values from 0 to 2³²-1. Both servers and keys are placed on this ring using their hash values.

```
         0
    ─────┬─────
  330    │    30
         │
 [S3]  ──┤──  [S1]
         │
  270    │    90
    ─────┴─────
        180
      [S2]
```

A key is served by the **first server clockwise** from the key's position on the ring.

**Adding a new server:** Only the keys between the new server's position and its predecessor need remapping. Typically ~1/N of total keys.

**Virtual Nodes:** To ensure even distribution, each physical server is represented by multiple virtual nodes on the ring (e.g., 150 virtual nodes per server).

**Used By:** Amazon DynamoDB, Apache Cassandra, Memcached, Discord's message routing.

```
   │                                            │
   └── GET /:code ──▶ [Redirect Service]        │
                           │                    │
                      [Redis Cache] ◀───────────┘
                      (cache short:long mapping)
                           │
                      [CDN Edge] ──▶ 301/302 Redirect
```

**Short Code Generation:**
```javascript
// Base62 encoding of auto-incremented ID
const BASE62 = '0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz';

function encodeBase62(num) {
  let result = '';
  while (num > 0) {
    result = BASE62[num % 62] + result;
    num = Math.floor(num / 62);
  }
  return result.padStart(7, '0'); // 7-character short code
}

// ID 1000000 → "041UKT"
```

### Design a Real-Time Chat System (like WhatsApp)

**Requirements:**
- One-on-one and group messaging
- Message delivery receipts (sent, delivered, read)
- Offline message delivery
- 50M concurrent users

**Key Design Decisions:**

*Connection Management:* WebSockets for real-time bidirectional communication.

```javascript
// Server — WebSocket connection management
const connections = new Map(); // userId → WebSocket

wss.on('connection', (ws, req) => {
  const userId = authenticate(req);
  connections.set(userId, ws);

  ws.on('message', async (rawMessage) => {
    const { recipientId, content } = JSON.parse(rawMessage);

    // Persist message
    const message = await db.messages.create({
      senderId: userId,
      recipientId,
      content,
      status: 'sent'
    });

    // Deliver in real-time if recipient is online
    const recipientWs = connections.get(recipientId);
    if (recipientWs) {
      recipientWs.send(JSON.stringify({ type: 'message', data: message }));
      await db.messages.updateStatus(message.id, 'delivered');
    } else {
      // Queue for offline delivery via Push Notification
      await pushQueue.publish({ userId: recipientId, messageId: message.id });
    }
  });

  ws.on('close', () => connections.delete(userId));
});
```

*Message Storage Schema:*
```sql
CREATE TABLE messages (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  sender_id    UUID NOT NULL REFERENCES users(id),
  recipient_id UUID NOT NULL REFERENCES users(id),
  content      TEXT NOT NULL,
  status       VARCHAR(20) DEFAULT 'sent',  -- sent | delivered | read
  created_at   TIMESTAMP DEFAULT NOW()
);

-- Index for fetching conversation history
CREATE INDEX idx_messages_conversation
ON messages (sender_id, recipient_id, created_at DESC);
```

## 16. System Design Interview Framework {#interview-framework}

A structured, repeatable approach for tackling any system design question.

### The RESHADE Framework

**R — Requirements Clarification (5 min)**
- "Should this support mobile and web?"
- "What's the expected DAU / MAU?"
- "Do we need real-time updates or is eventual consistency acceptable?"
- "What are the read vs write ratios?"

**E — Estimation (3 min)**
```
Daily Active Users (DAU): 10M
Requests per second (RPS): DAU × avg_requests_per_day / 86,400
Storage per year: write_RPS × payload_size × 86,400 × 365
Bandwidth: RPS × avg_response_size
```

**S — System Interface (2 min)**

Define the core API endpoints before drawing boxes:
```
POST /api/tweets          { content, mediaUrls }
GET  /api/timeline/:userId
POST /api/tweets/:id/like
```

**H — High-Level Design (10 min)**

Draw the 30,000-foot view: clients, load balancers, services, databases, cache. No deep-diving yet.

**A — Detailed Component Design (15 min)**

Pick the 2–3 most interesting or challenging components and go deep:
- How does the database schema look?
- How does the cache invalidation work?
- How do you handle race conditions?

**D — Data Store Selection (5 min)**

Justify your database choices explicitly:
- "I'd use PostgreSQL for user profiles because data is relational and ACID compliance matters for billing."
- "I'd use Cassandra for the activity feed because it's append-heavy with 10B writes/day and eventual consistency is acceptable."

**E — Edge Cases and Bottlenecks (5 min)**

Proactively identify failure modes:
- "The hot shard problem — if one celebrity's post goes viral, how do we prevent their shard from becoming overwhelmed?"
- "Clock skew in distributed systems — how do we order events across regions?"

## Final Architecture Checklist

Before calling any system design complete, validate against this checklist:

```
✅ Single points of failure eliminated (replicas, failover)
✅ Horizontal scalability path defined for all stateful components
✅ Caching strategy documented for hot data paths
✅ Database indexes cover all critical query patterns
✅ Rate limiting applied at API gateway and service level
✅ Async processing via queues for non-critical write paths
✅ Health checks and circuit breakers on all inter-service calls
✅ Data backup and point-in-time recovery strategy in place
✅ CDN configured for all static and publicly cacheable content
✅ Monitoring, alerting, and distributed tracing instrumented
```
