# 🚀 Docker: Enterprise Containerization Reference

Docker is an open-source platform engineered for building, running, and managing applications by packaging them into standardized, portable execution units called **containers**. The platform is architecturally composed of two primary primitives: **Docker Images** and **Docker Containers**.

---

## Core Concepts

### Docker Container

A standardized, executable unit of software that packages application code alongside all its runtime dependencies — ensuring the application executes quickly and reliably across heterogeneous computing environments.

---

### Docker Images

- A read-only template containing the application code, runtime, system tools, libraries, and dependencies required to instantiate a running application.
- Images are immutable filesystem snapshots; containers are their live, writable instantiations.

## Detached Mode Container Execution

The `-d` flag executes a container in **detached mode**, running the process in the background and returning control to the host shell immediately.

> **Default Behavior:** Without `-d`, all containers execute in **attached mode**, binding the terminal to the container's stdout/stderr.

The following example launches a MySQL container in detached mode, supplying a mandatory root password via the `-e` (environment variable) flag. For full MySQL container configuration documentation, refer to [https://hub.docker.com/_/mysql](https://hub.docker.com/_/mysql).

```bash
docker run -d -e MYSQL_ROOT_PASSWORD=secret mysql
```

![MySQL Container Running in Detached Mode — Terminal Output](/img/Docker/image-6.png)

![MySQL Container Visible in Docker Desktop Dashboard](/img/Docker/image-7.png)

### Named Container Instantiation

Assign a deterministic, human-readable name to a container at runtime:

```bash

```

## Docker Image Layers and Layer Caching

Docker images employ a **layered filesystem architecture** in which each layer represents an incremental filesystem delta. This design delivers three primary engineering benefits:

- **Storage Efficiency:** Shared base layers between image versions are stored once and referenced multiple times.
- **Accelerated Build Times:** Unchanged layers are served from cache during subsequent builds.
- **Simplified Version Control:** Each layer is independently addressable and reusable.

When pulling two versions of the same image, the CLI output will indicate which layers already exist locally and will be served from cache rather than re-downloaded.

## Port Binding

Port binding exposes container-internal network ports to the host machine, enabling external access to containerized services.

**Syntax:** `-p HOST_PORT:CONTAINER_PORT`

Bind host port `8080` to MySQL container port `3306`:

```bash
docker run -d -e MYSQL_ROOT_PASSWORD=secret --name mysql-latest -p8080:3306 mysql
```

![Port Binding — MySQL Latest on Host Port 8080](/img/Docker/image-8.png)

> **Constraint:** A given host machine port can only be bound to a single container at a time. Attempting to bind the same host port to a second container will result in an error.

```bash
docker run -d -e MYSQL_ROOT_PASSWORD=secret --name mysql-older -p8080:3306 mysql:8.0
```

![Port Conflict Error — Duplicate Host Port Binding Attempt](/img/Docker/image-9.png)

> **Resolution:** Multiple containers may expose the same internal container port, but each must be mapped to a **distinct host machine port**.

## Docker vs. Virtual Machines: Architectural Comparison

| Dimension | Docker | Virtual Machine |
|---|---|---|
| Virtualization Scope | Application layer only (shares host OS kernel) | Full OS kernel + Application layer |
| Resource Footprint | Lightweight | Heavy |
| Startup Time | Milliseconds | Seconds to minutes |
| Isolation Level | Process-level | Hardware-level |

- **Docker** achieves efficiency by virtualizing only the application layer, leveraging the host OS kernel directly.
- **Virtual Machines** virtualize both the OS kernel and the application layer, providing stronger isolation at higher resource cost.
- **Docker Desktop** ships with an embedded lightweight Linux-based VM to enable container execution on macOS and Windows host systems.

```bash
  -e ME_CONFIG_MONGODB_ADMINPASSWORD=qwerty \
```

## Docker Compose

Docker Compose is a declarative orchestration tool for defining and running multi-container applications. Configuration is expressed in a `.yaml` file.

> **YAML Definition:** *YAML Ain't Markup Language* — a human-readable data serialization standard.

**Key Advantages:**
- Standardized, reproducible command interface for multi-container stacks.
- Simplified environment modification and configuration management.

> **Critical Syntax Requirement:** YAML files are whitespace-sensitive. Correct indentation is mandatory for successful parsing.

![Docker Compose YAML File Structure Example](/img/Docker/e.png)

### Compose Lifecycle Commands

Start all defined services in detached mode:

```bash
docker compose -f fileName.yaml up -d
```

Stop all services and remove associated containers and networks:

```bash
docker compose -f fileName.yaml down
```

## Dockerizing an Application — Image Build Pipeline

The Dockerfile-driven build pipeline converts application source code into a portable, distributable Docker image.

```
Application Source → Dockerfile → Docker Image → Docker Container
```

A **Dockerfile** is a plain-text instruction manifest defining the sequential build steps for constructing a Docker image.

For comprehensive Dockerfile syntax reference, consult: [https://spacelift.io/blog/dockerfile](https://spacelift.io/blog/dockerfile)

---

### Essential Dockerfile Instruction Reference

**`FROM`** — Specifies the base image serving as the foundational layer for the new image:

```dockerfile
FROM <base_image>
```

**`WORKDIR`** — Establishes the working directory within the container for all subsequent `RUN`, `COPY`, `CMD`, and related instructions:

```dockerfile
WORKDIR <path>
```

**`COPY`** — Transfers files and directories from the local build context into the image filesystem:

```dockerfile
COPY <src> <dest>
```

**`RUN`** — Executes shell commands during the image build phase; results are committed as a new filesystem layer:

```dockerfile
RUN <command>
```

**`CMD`** — Declares the default executable and arguments invoked when a container is started without explicit command-line override:

```dockerfile
CMD <command>
```

**`EXPOSE`** — Documents the network port(s) on which the containerized application listens at runtime. Functions as operational documentation rather than an active port-publishing mechanism:

```dockerfile
EXPOSE <port>
```

**`ENV`** — Injects persistent environment variables into the container runtime context:

```dockerfile
ENV <name> <value>
```

## Docker Volumes — Persistent Data Management

By default, all data written inside a container is **ephemeral** — it is lost when the container is removed or Docker Desktop is restarted. **Docker Volumes** provide a mechanism for persistent, container-independent data storage.

**Key Characteristics:**

- **Persistent Storage:** Data stored in a volume survives container restarts and deletions.
- **Docker-Managed:** Volumes are created, managed, and stored by Docker, typically at `/var/lib/docker/volumes` on Linux hosts.
- **Declarative Mount Points:** Data written to the mounted container directory path is transparently persisted to the volume.
- **Multi-Container Sharing:** Multiple containers can concurrently read from and write to the same volume.
- **Backup and Recovery:** Volume isolation simplifies data backup, migration, and disaster recovery operations.
- **Named vs. Anonymous Volumes:**
  - **Named Volumes** are explicitly user-defined and are the recommended approach for production workloads.
  - **Anonymous Volumes** are automatically generated and are tied to the lifecycle of their originating container.
- **Volume Drivers:** Enable storing volumes on remote hosts or cloud-based storage providers.

### Bind Mount — Mapping Host Directory to Container

The `-v` flag specifies volume configuration. The following command mounts a local host directory into the container at a specified path:

```bash
docker run -it -v /Users/RUDRANARAYAN/Desktop/data:/test/data ubuntu
```

> **Note:** Replace `RUDRANARAYAN` with your system's hostname or username.

This establishes a **bind mount**, where the specified host directory is directly mapped to the container directory. Changes in either location are reflected in both.

![Bind Mount — Host Directory Mapped to Ubuntu Container Path](/img/Docker/image-27.png)

### Container Filesystem Interaction via Bind Mount

Inside the container, navigate to the mounted directory and create test assets:


## Docker Volume CLI Reference

List all managed volumes:

```bash
docker volume ls
```

Create a named volume:

```bash
docker volume create VOL_NAME
```

Remove a named volume:

```bash
docker volume rm VOL_NAME
```

**Named Volume** (Docker-managed, persistent):

```bash
docker run -v VOL_NAME:CONT_DIR
```

**Bind Mount** (Host-managed, maps a specific host path):

```bash
docker run -v HOST_DIR:CONT_DIR
```

**Anonymous Volume** (Docker-managed, container-lifecycle-scoped):

```bash
docker run -v MOUNT_PATH
```

Purge all dangling anonymous volumes:

```bash
docker volume prune
```
